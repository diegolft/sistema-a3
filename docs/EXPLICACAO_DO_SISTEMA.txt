  GAME STORE — EXPLICAÇÃO DO SISTEMA PARA O GRUPO
  (Escrita para quem conhece HTML, CSS e JS básico)
============================================================

Se você já sabe criar uma página HTML, estilizar com CSS e fazer

------------------------------------------------------------
1. O QUE É ESSE PROJETO?
------------------------------------------------------------

É uma loja virtual de jogos digitais. Pense no Steam ou na
PlayStation Store, mas menor e feita por nós para o trabalho.

O usuário pode:
  - Criar conta e fazer login
  - Ver o catálogo de jogos
  - Adicionar jogos ao carrinho e comprar
  - Ver sua biblioteca de jogos comprados
  - Avaliar os jogos com nota e comentário
  - Ver o histórico de compras

O administrador também pode:
  - Gerenciar jogos, empresas e categorias
  - Ver relatórios de vendas com gráficos


------------------------------------------------------------
2. TECNOLOGIAS USADAS (e o que cada uma faz)
------------------------------------------------------------

REACT
  O que você conhece: HTML é escrito direto no arquivo .html.
  O que o React faz: Deixa você escrever o HTML dentro do
  JavaScript, em arquivos .tsx. Cada "componente" é uma função
  que retorna HTML. Quando os dados mudam, o React atualiza
  a tela sozinho, sem precisar de document.getElementById().

  Exemplo simples:
    function BotaoOla() {
      return <button>Olá!</button>;
    }
  Isso é React. Parece HTML dentro do JS — isso se chama JSX.

TYPESCRIPT (.tsx em vez de .jsx)
  É JavaScript com "tipos". Ajuda a evitar erros. Por exemplo,
  se uma função espera um número e você passa uma string,
  ele avisa antes de rodar. Pode ignorar os ": string", ": number"
  que aparecem no código — é só isso.

VITE
  É o programa que "compila" o projeto e sobe um servidor local.
  Equivale ao "abrir o index.html no navegador", mas com muito
  mais recursos. Comando para rodar: bun dev (ou npm run dev).

TAILWIND CSS
  Em vez de criar classes no CSS separado, você coloca as classes
  diretamente no HTML. Por exemplo:
    <div class="flex gap-4 text-white bg-black p-5">
  Cada palavra é uma regra CSS pronta. É como CSS inline, mas
  organizado e responsivo.

REACT ROUTER DOM
  Cuida da navegação entre páginas SEM recarregar o navegador.
  Quando você clica em "Jogos", a URL muda para /jogos mas a
  página NÃO recarrega — o React apenas troca o componente
  exibido. É como trocar uma seção de uma SPA (Single Page App).

FRAMER MOTION
  Biblioteca de animações. Usamos para o fade e deslize suave
  quando muda de página, e para os cards de jogos.

RECHARTS
  Biblioteca de gráficos. Usada na página de relatórios do admin
  para mostrar barras e tortas com dados de vendas.

SONNER
  Biblioteca de notificações (os "toasts" — aquelas mensagens
  que aparecem no canto da tela). Ex: "Adicionado ao carrinho!".

LUCIDE REACT
  Pacote de ícones SVG. O ícone de carrinho, coração, lixeira,
  etc. todos vêm daqui.

------------------------------------------------------------
4. COMO O REACT "MONTA" A TELA (fluxo básico)
------------------------------------------------------------

Passo a passo do que acontece quando você abre o site:

  1. O navegador carrega index.html (tem só uma <div id="root">)
  2. main.tsx injeta o React dentro dessa div
  3. App.tsx entra em ação e lê a URL atual
  4. O React Router decide qual componente/página mostrar
  5. O componente da página faz chamadas à API para buscar dados
  6. Os dados chegam e o React atualiza a tela automaticamente

Você NUNCA precisa escrever document.getElementById() ou
innerHTML = "...". O React faz isso internamente.


------------------------------------------------------------
5. O QUE SÃO "CONTEXTS"? (estado global)
------------------------------------------------------------

Imagine que você quer mostrar o número de itens do carrinho
no header E na página do carrinho ao mesmo tempo.

Com JS puro, você precisaria de variáveis globais ou ficar
passando dados entre funções manualmente — vira bagunça.

O React Context resolve isso: é uma "caixa de dados" que
qualquer componente pode acessar, não importa onde esteja
na árvore de componentes.

  AuthContext   → Guarda: o usuário logado, o token JWT,
                  se é admin. Qualquer página pode perguntar
                  "o usuário está logado?" sem receber isso
                  como parâmetro.

  CartContext   → Guarda: os itens do carrinho e o total.
                  O Header usa para mostrar o badge (número
                  de itens). A CartPage usa para listar tudo.

  LibraryContext → Guarda: os jogos que o usuário comprou.
                  Usado na LibraryPage e no GameDetailPage
                  para mostrar "Jogo já na biblioteca".

Todos os Contexts ficam em src/contexts/ e são "embrulhados"
em volta de toda a aplicação no App.tsx.


------------------------------------------------------------
6. COMO FUNCIONA A COMUNICAÇÃO COM O BACKEND (API)
------------------------------------------------------------

O backend é um servidor Java separado. Ele tem uma URL base,
por exemplo: http://localhost:8080/api

O frontend conversa com ele usando fetch() — que você já
conhece do JS básico. Só que aqui usamos async/await para
deixar o código mais limpo.

Exemplo simplificado de como buscamos os jogos:

  async function buscarJogos(token) {
    const resposta = await fetch("http://localhost:8080/api/jogos", {
      headers: {
        "Authorization": "Bearer " + token
      }
    });
    const dados = await resposta.json();
    return dados;
  }

O "Bearer token" é o JWT (veja seção 7).

Todos esses fetches ficam organizados em src/services/api/.
Cada arquivo cuida de uma parte da API:
  - games.ts    → /api/jogos
  - auth.ts     → /api/auth/login, /api/auth/register
  - cart.ts     → /api/carrinho
  - reviews.ts  → /api/avaliacoes
  - reports.ts  → /api/relatorios


------------------------------------------------------------
7. O QUE É JWT E COMO FUNCIONA O LOGIN?
------------------------------------------------------------

JWT = JSON Web Token. É como um "crachá digital".

Quando o usuário faz login (email + senha), o backend verifica
os dados e, se estiver tudo certo, devolve um token — uma
string longa, tipo:

  eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyQGV4YW1wbGUu...

O frontend guarda esse token (no localStorage do navegador)
e inclui ele em TODA chamada à API no cabeçalho:
  Authorization: Bearer <token>

O backend usa o token para saber QUEM está fazendo a requisição
e SE tem permissão. Sem o token, o backend recusa (erro 401).

No nosso código:
  - Login → AuthContext guarda o token
  - Logout → AuthContext apaga o token
  - Proteção de rotas → RequireAuth.tsx checa se tem token


------------------------------------------------------------
8. PROTEÇÃO DE ROTAS
------------------------------------------------------------

Algumas páginas só podem ser acessadas por usuários logados.
Outras só por administradores. Isso é feito com dois componentes:

  RequireAuth   → Se não estiver logado, redireciona para /login
  RequireRole   → Se não for admin, redireciona para /jogos

No App.tsx você vê isso assim:

  <Route element={<RequireAuth />}>
    <Route path="carrinho" element={<CartPage />} />
    <Route path="biblioteca" element={<LibraryPage />} />
  </Route>

  <Route element={<RequireRole role="Administrador" />}>
    <Route path="admin" element={<AdminLayout />}>
      ...
    </Route>
  </Route>

Qualquer rota "dentro" do RequireAuth só abre se logado.

9. TODAS AS PÁGINAS E O QUE CADA UMA FAZ
------------------------------------------------------------

PÚBLICAS (qualquer pessoa vê, sem login):
  /              → Landing page com vitrine de jogos em destaque
  /jogos         → Catálogo completo com busca e filtro por categoria
  /faq           → Perguntas frequentes com accordion animado
  /termos        → Termos de uso
  /privacidade   → Política de privacidade

  /cadastro      → Formulário de cadastro
  /esqueci-senha → Solicitar redefinição de senha
  /alterar-senha → Formulário de nova senha

PRIVADAS (precisam de login):
  /jogos/:id     → Detalhe do jogo + formulário de avaliação
  /carrinho      → Carrinho com itens, método de pagamento e checkout
  /lista-desejos → Jogos salvos para comprar depois
  /biblioteca    → Jogos já comprados, com chave de ativação
  /historico     → Histórico de todas as compras realizadas
  /perfil        → Editar nome, data de nascimento e senha

ADMIN (precisam de login + perfil Administrador):
  /admin/relatorios  → Gráficos: top vendidos, por empresa, por categoria
  /admin/usuarios    → Listar e editar usuários
  /admin/empresas    → CRUD de publishers (empresas)
  /admin/perfis      → Perfis e papéis disponíveis
  /admin/jogos       → Cadastrar e editar jogos do catálogo
  /admin/categorias  → Listar categorias de jogos

10. ACESSIBILIDADE — O QUE IMPLEMENTAMOS E POR QUÊ
------------------------------------------------------------

O trabalho exige critérios de acessibilidade. Veja o que foi feito:

  - Navegação por teclado: todos os botões e links funcionam
    com Tab e Enter. Os ícones têm aria-label para leitores
    de tela saberem o que são.

  - HTML semântico: usamos <header>, <nav>, <main>, <footer>,
    <ul>/<li>, <article>, <time>, <fieldset>/<legend>,
    <section> com labels. Isso ajuda leitores de tela (como
    o NVDA) a entender a estrutura da página.

  - Contraste: fundo escuro (#121212) com texto claro
    (neutral-100 = quase branco). O laranja de destaque
    (#ff7a00) é usado só em elementos importantes.

  - role="search" na busca de jogos
  - role="group" + aria-label no filtro de categorias
  - aria-expanded nos botões do FAQ (accordion)
  - aria-label em todos os ícones sem texto visível

11. HEURÍSTICAS DE NIELSEN IMPLEMENTADAS
------------------------------------------------------------

O trabalho exige no mínimo 5. Veja onde cada uma aparece:

  1. Visibilidade do status
     → Loading skeletons mostram que dados estão carregando.
     → Notificações (toasts) confirmam cada ação do usuário.
     → Badge no ícone do carrinho mostra quantos itens tem.

  2. Correspondência com o mundo real
     → Ícones reconhecíveis: carrinho, coração, lixeira.
     → Linguagem em português, sem jargões técnicos.
     → Preços em R$ com vírgula, no formato brasileiro.

  3. Controle e liberdade do usuário
     → Botão "Desfazer" aparece ao remover item do carrinho.
     → Dialog de confirmação antes de finalizar compra.
     → Botão "←" para voltar na página de detalhe do jogo.

  4. Consistência e padrões
     → Todos os botões de ação têm o mesmo estilo laranja.
     → Todos os cards de jogo têm o mesmo layout.
     → O header e footer aparecem em todas as páginas.

  5. Prevenção de erros
     → Validação de formulários antes de enviar.
     → Confirmação antes de sair da conta e antes de comprar.
     → Campos desabilitados quando uma ação está em andamento.

  6. Reconhecimento em vez de memorização
     → Navegação sempre visível no topo.
     → Estado ativo destacado nos links de navegação.
     → Ícones + texto nos botões principais.

  7. Flexibilidade e eficiência
     → Atalho de teclado Alt+H abre o FAQ de qualquer página.
     → Busca + filtro por categoria na página de jogos.
     → Filtro de Top N nos relatórios do admin.

  8. Design estético e minimalista
     → Interface escura e limpa, sem elementos desnecessários.
     → Informações secundárias em cinza, principais em branco.

12. COMO RODAR O PROJETO LOCALMENTE
------------------------------------------------------------

Pré-requisitos:
  - Node.js instalado (ou Bun — preferido)
  - Backend Java rodando na porta 8080

Passos:
  1. Abra o terminal na pasta do projeto (A3web)
  2. Execute: bun install   (ou: npm install)
  3. Execute: bun dev       (ou: npm run dev)
  4. Abra no navegador: http://localhost:5173

O banco de dados precisa estar rodando antes.
Comando SQL para criar: CREATE DATABASE avjd;
(O backend configura as tabelas automaticamente via JPA)

13. GLOSSÁRIO RÁPIDO
------------------------------------------------------------

Componente     → Uma função React que retorna HTML (JSX)
Props          → Parâmetros passados para um componente
State (useState) → Variável que, quando muda, atualiza a tela
Hook           → Função especial do React (começa com "use")
Context        → Dados compartilhados globalmente
useEffect      → Executa código quando o componente carrega
                 (equivale ao window.onload do JS puro)
async/await    → Jeito moderno de fazer fetch (chamadas à API)
JWT            → Token de autenticação devolvido pelo login
SPA            → Single Page Application — o site nunca
                 recarrega, só troca os componentes na tela
Vite           → Ferramenta que compila e serve o projeto
Tailwind       → CSS via classes no próprio HTML
TypeScript     → JavaScript com tipagem (os ": string" etc.)
.tsx           → Arquivo TypeScript com JSX (HTML no JS)
Bundle         → O arquivo final gerado pelo "bun build",
                 que vai para produção
